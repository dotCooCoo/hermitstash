# Changelog

One entry per released tag, grouped by minor. Latest first.

Entries are generated from `release-notes/v<version>.json`. The full
change history before v1.11.7 is available on the GitHub Releases page.

## v1.11.x

- v1.11.9 (2026-05-21) ÔÇö **Tag-verify gate now fetches `origin/main` before walking first-parent.** The integrity gate added in v1.11.7 walks `git rev-list --first-parent origin/main` to confirm a tagged commit landed on the main branch. On a tag-push event, `actions/checkout@v6` checks out the tag's commit and leaves `origin/main` either unset or stale, so the rev-list grep returned no match and the gate failed every release with a misleading 'not on main' error. The fix adds an explicit `git fetch --no-tags origin main:refs/remotes/origin/main` before the walk. **Fixed:** *`sha-to-tag-verify.yml` ÔÇö explicit `origin/main` fetch* ÔÇö The workflow now refreshes the `origin/main` remote-tracking ref before the first-parent walk, so the rev-list reflects the actual current state of the branch the tag was pushed against. The job's step name was also updated to drop the stale 'merged via PR' wording (the PR-merge check was removed in v1.11.8).

- v1.11.8 (2026-05-21) ÔÇö **Release pipeline hot-fix ÔÇö version resolution + tag-verify gate.** Fixes two release-pipeline regressions from v1.11.7. The version-resolution step in the Docker workflow silently fell back to `dev` on a node error, producing `hermitstash-vdev.image.tar.*` sidecars that the release job couldn't match against the expected `hermitstash-v1.11.7.image.tar.*` paths ÔÇö the GitHub Release for v1.11.7 created with an empty asset list as a result. The sha-to-tag-verify gate also required every tagged commit to be the result of a PR merge, but HermitStash uses direct-to-main pushes on both private and public repos (no PR squash-merge flow), so the gate failed every release. **Fixed:** *`docker-publish.yml` version resolution* ÔÇö On tag pushes the version is now derived directly from `GITHUB_REF` (`refs/tags/vX.Y.Z` Ôćĺ `X.Y.Z`) and cross-checked against `lib/constants.js`; a mismatch fails the workflow loud. The prior shape ran `node -e "...constants.version" 2>/dev/null || echo "dev"` which masked a real error and silently produced `vdev`-named artifacts. Workflow-dispatch (manual) runs still fall back to `lib/constants.js` since there is no tag context. ┬Ě *`sha-to-tag-verify.yml` PR-merge requirement removed* ÔÇö The gate now verifies the tag SHA is on `main`'s first-parent history AND the tag object is annotated + SSH-signed against the maintainer key ÔÇö the two checks that actually defend against the tag-mutation class (CVE-2025-30066) and source-side malicious-publish class (TanStack 2025-05-11). The PR-merge requirement was ported verbatim from a PR-flow upstream but does not fit this project's direct-push release flow. **References:** [CVE-2025-30066 (tj-actions tag mutation)](https://nvd.nist.gov/vuln/detail/CVE-2025-30066) ┬Ě [SLSA v1.0 specification](https://slsa.dev/spec/v1.0/)

- v1.11.7 (2026-05-21) ÔÇö **Structured release notes + SLSA L3 container provenance + ML-DSA-65 release-signing sidecar.** Releases now ship with a SLSA L3 provenance attestation bound to the workflow run + tag + commit, an ML-DSA-65 PQC signature sidecar over the saved image tarball, and SHA-256 + SHA3-512 byte-level digest sidecars. A new tag-push gate refuses tags that aren't on main's first-parent + merged via a PR + SSH-signed against the maintainer key. Release notes move to a structured `release-notes/v<version>.json` source with a leak-vocabulary validator; the GitHub Release page is generated from that source by the workflow at tag-push time. **Added:** *SLSA L3 provenance attestation on the OCI image* ÔÇö The new `provenance` job in `docker-publish.yml` invokes `slsa-framework/slsa-github-generator/.github/workflows/generator_container_slsa3.yml@v2.1.0` against the pushed image digest. The container generator signs the attestation via Sigstore-keyless and pushes it to GHCR alongside the image. Operators verify with `slsa-verifier verify-image ghcr.io/dotcoocoo/hermitstash@<digest> --source-uri github.com/dotCooCoo/hermitstash --source-tag vX.Y.Z`. ┬Ě *ML-DSA-65 release-signing sidecar (`<image>.mldsa.sig`)* ÔÇö `scripts/sign-release-artifact.js` signs the `docker save` tarball bytes with the project's ML-DSA-65 release-signing key (FIPS 204, NIST PQC level 3) and writes the signature alongside. The signer self-verifies against the in-tree public key (`keys/release-pqc-pub.json`) before writing, refusing to ship an unverifiable sidecar when the secret drifts from the committed pubkey. Skips gracefully with a warning when the operator has not yet completed the one-time key-setup. ┬Ě *SHA-256 + SHA3-512 digest sidecars (`<image>.sha256`, `<image>.sha3-512`)* ÔÇö `scripts/sha3-digest.js` writes both digests in `sha256sum -c` / `sha3sum -a 512`-compatible format. Operators verify the saved-image bytes against the published sidecars without trusting the registry view: `sha256sum -c hermitstash-vX.Y.Z.image.tar.sha256` and `openssl dgst -sha3-512 hermitstash-vX.Y.Z.image.tar`. ┬Ě *Structured release notes ÔÇö `release-notes/v<version>.json`* ÔÇö Per-version JSON files are the single source of truth for the changelog. `scripts/generate-changelog-entry.js` validates each entry against the section allowlist (Added / Changed / Deprecated / Removed / Fixed / Security / Migration), the title/body format rules, and a leak-vocabulary sweep, then emits either the one-line CHANGELOG.md bullet or the multi-section GitHub Release page markdown. `--rebuild` regenerates the full CHANGELOG.md from the JSON tree; `--check` is a non-mutating drift gate. ┬Ě *`scripts/check-changelog-extract.js` ÔÇö pre-push CHANGELOG gate* ÔÇö Exercises the awk-extract path the release workflow uses to pull the current version's CHANGELOG section into the release-page markdown. Running it locally before tag-push catches format drift (e.g. a new entry that doesn't match the canonical `- vX.Y.Z (YYYY-MM-DD) ÔÇö <summary>` shape) without burning a CI run. ┬Ě *`.github/workflows/sha-to-tag-verify.yml` ÔÇö tag-push integrity gate* ÔÇö Refuses tags whose commit SHA is not on `main`'s first-parent history, was not merged via a pull request, or whose tag object is not annotated + SSH-signed against the maintainer key (`ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAAIPiE/PETpyiVPd8aMygJ+S9CsSVolp4HQZaAuiYVwbBa`). Defends against tag-mutation (CVE-2025-30066 `tj-actions/changed-files` class) and source-side-malicious-publish (TanStack 2025-05-11 class) by verifying the chain at the workflow boundary independently of the GitHub edge ruleset. ┬Ě *Atomic release creation with full asset bundle* ÔÇö The new `release` job in `docker-publish.yml` calls `gh release create` once with the digest sidecars, the ML-DSA-65 sig sidecar (when available), and the CSAF 2.1 VEX document attached at creation time. Refuses to overwrite an existing release for the same tag ÔÇö the workflow is now authoritative for release creation; the manual `gh release create` step is removed from the release playbook. ┬Ě *Codebase-patterns detectors ÔÇö dense-wildcard / uncapped-searchparams-object / hex-sha-compare-equals / inline-require-in-deferred* ÔÇö Four new lint gates ported from the upstream framework. `dense-wildcard` refuses 4+ consecutive `*` in lib/ (CVE-2026-4923 catastrophic-backtracking glob class). `uncapped-searchparams-object` flags `Object.fromEntries(searchParams)` without a key cap (CVE-2026-21717 V8 HashDoS). `hex-sha-compare-equals` flags `hmac*Hex` / `mac*Hex` / `signature*Hex` compared with `===` / `!==` (CVE-2026-21713 timing-leak). `inline-require-in-deferred` flags `require()` inside `setImmediate` / `process.nextTick` / `queueMicrotask` (CRYPTO-15 module-load race). **Changed:** *Vendored blamejs framework: v0.11.2 Ôćĺ v0.11.30* ÔÇö 28 patches forward.
