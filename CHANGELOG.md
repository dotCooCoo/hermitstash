## v1.11.x

- v1.11.26 (2026-05-31) — **Vault-key rotation re-seals row-bound values; actionable restore errors.** Vault-key rotation again works on deployments that hold row-bound sealed values, and a failed restore now returns an actionable status code instead of a generic 500. **Changed:** *Re-seal row-bound values during vault-key rotation* — Vault-key rotation now re-seals the AEAD row-bound sealed values under the new key and verifies that the old key can no longer open them. The earlier safeguard that refused to rotate while such values were present — added when row-bound sealing shipped — is removed, so rotation works again on any deployment. · *Update the vendored framework to blamejs 0.14.13* — Refreshes the vendored blamejs framework from 0.14.11 to 0.14.13, which adds the explicit-root re-seal the rotation tool relies on for row-bound values. **Fixed:** *Return an actionable status code from a failed restore* — A failed backup restore previously reported a generic 500 with no detail. A missing backup now returns 404, a wrong passphrase or undecryptable archive returns 403, and a failed integrity check returns 422 — each with an operator-facing message — while genuinely internal failures remain a 500 with the detail suppressed.

- v1.11.25 (2026-05-31) — **Problem-details errors across the account and admin endpoints.** Authentication, account, and admin endpoints now return their failures as RFC 9457 problem-details instead of the legacy { error } body, extending the format introduced in v1.10.1 to the rest of the surface. **Changed:** *Return problem-details from the account and admin endpoints* — Authentication, profile, user and team management, two-factor, passkeys, the encrypted vault, the admin panel, API keys, webhooks, password reset, and certificate enrollment now emit RFC 9457 problem-details ({ type, title, status, detail }) on failure, matching the format the rest of the API already used. 5xx detail stays suppressed so internal failure text never reaches clients; 4xx carries the actionable message. · *Update the vendored framework to blamejs 0.14.11* — Refreshes the vendored blamejs framework from 0.14.10 to 0.14.11. **Fixed:** *Return a real status code for failed admin backup and storage actions* — A handful of admin backup and storage-connection actions reported failure with an HTTP 200 body; they now return the appropriate 4xx or 5xx status. **Migration:** *Read the problem detail field for error messages* — API clients that read the legacy { error } field for a failure message should read { detail }, using { type } as the stable problem identifier. The web UI was updated in the previous release. Reading detail and falling back to error handles every endpoint, including the upload and sync endpoints that still return the legacy shape.

- v1.11.24 (2026-05-31) — **Surface problem-details error text in the web UI.** The web UI now reads RFC 9457 problem-details error messages, so failed actions show the server's reason instead of a blank message. **Fixed:** *Show problem-details error messages in the web UI* — Many web UI actions read only the legacy { error } field from a failed response, so when the server replied with an RFC 9457 problem-details body the on-screen message could be blank. The UI now reads the problem detail and falls back to the legacy field, and several actions that previously showed no message on failure (login, registration, user and stash management, and others) now display the server's reason.

- v1.11.23 (2026-05-31) — **Encrypt error responses on payload-encrypted sessions.** Error responses on sessions that use API payload encryption now travel inside the same encryption envelope as successful responses, instead of as cleartext problem-details. **Security:** *Encrypt API error responses on payload-encrypted sessions* — On a session that uses API payload encryption (browser sessions), successful responses were already encrypted but error responses were emitted as cleartext RFC 9457 problem-details. They are now delivered inside the same encryption envelope as successful responses, keeping the error body and its integrity within the encrypted channel. This matters most on HTTP-mode deployments where the payload-encryption layer is the only on-wire protection. Clients that already decrypt successful responses need no change: the decrypted body is the same problem-details object, with the HTTP status preserved.

- v1.11.22 (2026-05-31) — **Conditional and range static delivery, hardened webhook transport.** Static assets now support conditional and partial requests, and outbound webhook delivery routes through the framework HTTP client for a single, well-tested egress path. **Changed:** *Serve static assets with conditional and range support* — Responses for built-in static assets (CSS, JavaScript bundles, fonts, images) now carry an ETag validator and honor If-None-Match with 304 Not Modified, plus RFC 7233 Range requests. Repeat page loads revalidate instead of re-downloading, and large assets can be fetched in parts. · *Update the vendored framework to blamejs 0.14.10* — Refreshes the vendored blamejs framework from 0.14.9 to 0.14.10. **Security:** *Route webhook delivery through the framework HTTP client* — Outbound webhook POSTs now go through the framework HTTP client. It re-checks the destination against the SSRF policy and pins the connection to the validated address, requires HTTPS, refuses to follow redirects, and caps both total and idle time. The X-Webhook-Signature (HMAC-SHA3-512) header, the per-delivery log, and the retry-with-backoff behavior are unchanged.

- v1.11.21 (2026-05-31) — **Row-bound sealing for the remaining sealed columns.** The TOTP secret, the per-file content-encryption key, and operator settings values now AEAD-bind to their row identity, completing the row-binding the other sealed columns received in the prior release. The request-id middleware moves to the framework primitive. **Changed:** *Adopt the framework request-id middleware* — The X-Request-Id middleware now uses the framework primitive: it honors a well-formed trusted upstream X-Request-Id and mints a longer random id otherwise, while preserving the per-request log-correlation context. · *Update the vendored framework to blamejs 0.14.9* — Refreshes the vendored blamejs framework from 0.14.8 to 0.14.9. **Security:** *Row-bind the remaining sealed columns* — The TOTP secret, the per-file content-encryption key, and operator settings values are now AEAD-bound to their (table, primary key, column) identity — the same protection the other sealed columns received in the prior release. A value copied into another row or column fails authentication and reads as absent instead of decrypting. Existing values read unchanged and are rewritten to the bound form the next time each row is written.

- v1.11.20 (2026-05-31) — **Row-bound sealed columns and problem-details for the sync endpoints.** Sealed database columns are now authenticated against their row identity, so a sealed value copied between rows or columns no longer decrypts. The remaining sync endpoints adopt RFC 9457 problem-details, and outbound host checks reject trailing-dot bypasses. **Changed:** *Return problem-details from the sync endpoints* — POST /sync/enroll, /sync/renew-cert, and /sync/rename now return application/problem+json (RFC 9457) on error, matching the rest of the API. A client that reads the legacy error field should read detail instead. · *Update the vendored framework to blamejs 0.14.8* — Refreshes the vendored blamejs framework from 0.14.4 to 0.14.8. **Security:** *Bind sealed database columns to their row identity* — Every sealed column is AEAD-bound to its table, primary key, and column name. A value copied into a different row or column fails authentication and reads as absent instead of decrypting, closing a database-write relocation attack. Existing sealed data reads unchanged and is rewritten to the bound form the next time each row is written. Vault-key rotation refuses to run while row-bound values are present, since rotation would otherwise leave them undecryptable; rotation support for bound values is pending a framework update. · *Reject trailing-dot SSRF host bypasses* — Outbound host allow and deny checks strip trailing root-zone dots (RFC 1034) before comparison, so a host such as localhost. can no longer slip past the loopback and cloud metadata-endpoint guards.

- v1.11.19 (2026-05-29) — **Application logs scrub secret-shaped fields by default.** Structured application log lines now pass their fields through a redactor before they are written, so password-, token-, and key-shaped values are stripped automatically. This release also refreshes the vendored framework to blamejs 0.13.43. **Changed:** *Refresh the vendored framework to blamejs 0.13.43* — Picks up the current framework crypto, logging, and middleware primitives. **Security:** *Redact secret-shaped fields from application logs by default* — Structured log lines now scrub password / token / key-shaped values automatically before they are written. Unicode bidi and format controls in log messages are escaped so a hostile value cannot reorder a log line in a terminal or syslog viewer.

- v1.11.18 (2026-05-27) — **Browser features that need a secure context degrade gracefully over plain HTTP.** On a plain-HTTP origin at a non-localhost hostname — a LAN host or a reverse-proxy deployment without TLS — several browser APIs are simply absent (WebAuthn, the async Clipboard API, crypto.randomUUID). HermitStash now detects their absence and adapts instead of throwing "Cannot read properties of undefined": the passkey option hides itself, copy-to-clipboard uses a legacy fallback, and multi-file uploads generate batch IDs without the secure-context API. **Fixed:** *Passkey sign-in and registration hide on non-secure origins* — The login and profile passkey controls now feature-detect the WebAuthn API and, when it is absent (any plain-HTTP non-localhost origin, and Safari before 16.4 even over HTTPS), hide with a short "Passkeys require a secure (HTTPS) connection" note instead of throwing on click. They reappear automatically once the app is served over HTTPS. The PASSKEY_ENABLED setting is unchanged. · *Copy-to-clipboard falls back to a legacy path over plain HTTP* — Share-link, stash-link, and enrollment-code copy buttons route through a shared helper that uses the async Clipboard API in a secure context and falls back to document.execCommand('copy') over plain HTTP, so copying works regardless of scheme. · *Multi-file uploads no longer depend on a secure-context UUID API* — Batch IDs for multi-file uploads fall back from crypto.randomUUID (secure-context only) to crypto.getRandomValues, which is available over plain HTTP. **References:** [MDN — Secure contexts](https://developer.mozilla.org/en-US/docs/Web/Security/Secure_Contexts)

- v1.11.17 (2026-05-26) — **Bot guard no longer blocks browsers that omit `Sec-Fetch-Mode`.** Browsers reaching the app over plain HTTP at a non-localhost hostname — LAN access and reverse-proxy deployments that terminate TLS upstream — omit the `Sec-Fetch-*` request headers, which per the Fetch Metadata spec are sent only to secure-context origins. The bot guard treated a missing `Sec-Fetch-Mode` as a non-browser signal and returned `Forbidden` on every page load in those setups. The header is now advisory only; genuine automation is still refused by the missing-`Accept-Language` check and the User-Agent deny-list. **Fixed:** *Bot guard demotes the `Sec-Fetch-Mode` heuristic to advisory* — The vendored `b.middleware.botGuard` no longer refuses a request solely because `Sec-Fetch-Mode` is absent. Browsers omit every `Sec-Fetch-*` header on a non-secure-context origin — a plain-HTTP LAN host such as `http://hermitstash.local:3080`, or any reverse proxy that terminates TLS upstream and forwards cleartext — and Safari omitted them even over HTTPS before 16.4. A missing `Sec-Fetch-Mode` now contributes only as an advisory signal; blocking decisions rest on the missing-`Accept-Language` check and the known-automation User-Agent deny-list (curl, wget, python-requests, …). · *`tests/unit/bot-guard.test.js`* — New unit test drives the bot-guard middleware with a request shaped like a browser behind a plain-HTTP reverse proxy (host header set, `Accept-Language` present, no `Sec-Fetch-Mode`) and asserts it reaches the app rather than a `403`, while a `curl` User-Agent and an `Accept-Language`-less request remain refused. **References:** [W3C Fetch Metadata Request Headers](https://www.w3.org/TR/fetch-metadata/)

- v1.11.12 (2026-05-21) — **Test-scoped codebase-patterns — sleep, hardcoded bind ports, DB isolation.** Adds three test-scoped antipatterns to the codebase-patterns catalog so the lint gate now scans `tests/*.test.js` and `tests/helpers/` in addition to `lib/`. - v1.11.10 (2026-05-21) — **Remove `sha-to-tag-verify` workflow — redundant with GitHub edge ruleset.** The `sha-to-tag-verify` workflow added in v1.11.7 attempted to verify, from inside CI, that each tagged commit was on `main`'s first-parent history and signed by the maintainer's SSH key. - v1.11.9 (2026-05-21) — **Tag-verify gate now fetches `origin/main` before walking first-parent.** The integrity gate added in v1.11.7 walks `git rev-list --first-parent origin/main` to confirm a tagged commit landed on the main branch. On a tag-push event, `actions/checkout@v6` checks out the tag's commit and leaves `origin/main` either unset or stale, so the rev-list grep returned no match and the gate failed every release with a misleading 'not on main' error. The fix adds an explicit `git fetch --no-tags origin main:refs/remotes/origin/main` before the walk. **Fixed:** *`sha-to-tag-verify.yml` — explicit `origin/main` fetch* — The workflow now refreshes the `origin/main` remote-tracking ref before the first-parent walk, so the rev-list reflects the actual current state of the branch the tag was pushed against. - v1.11.8 (2026-05-21) — **Release pipeline hot-fix — version resolution + tag-verify gate.** Fixes two release-pipeline regressions from v1.11.7. The ciphertext leaves the server in the same form it sits on disk. · *UI refresh across admin and drop surfaces* — User-management buttons gained icons and color coding, the Danger Zone was redesigned with a card layout, the Customer Stash admin panel uses a sectioned form, the drop page logo has an animated glow, stash mode drops step numbers and name/email fields, the reverse-proxy config block is collapsed by default, and the GitHub and Support links use shimmer animations. **Fixed:** *Router middleware regression breaking all uploads* — The custom router invoked route-level middleware as `(req, res)` with no `next` callback, producing `TypeError: next is not a function` on every upload since v1.0.2 and surfacing as upload 500 errors. The router now detects 3-arg handlers and passes a real `next` callback. · *Vault enable flow now uses ML-KEM-1024* — The enable flow had drifted from the rotation flow; both now use ML-KEM-1024, aligning with FIPS 203 and matching the on-disk envelope produced by rotation. · *Vault rotation correctness* — Four rotation bugs resolved: base64url encoding on the seed transport, 64-byte seed generation, server-side seed validation, and zero-padded seed expansion. Without these, rotated vaults could produce keys that failed to decrypt previously sealed material. · *Vault file download seed expansion* — `expandSeed` now zero-pads to match the dashboard enable flow. Files written by the enable flow are decryptable by the download path again. · *PDF magic-byte detection per Adobe spec* — Detection scans the first 1024 bytes for `%PDF-` rather than only the first four, correctly handling files with a BOM or leading whitespace as the Adobe spec allows. · *ZIP detection accepts OOXML variants* — All PK signature variants are accepted so `.docx`, `.xlsx`, and `.pptx` uploads pass content validation. · *Stash stats decrement on every delete path* — Bundle and file deletion now adjust per-stash bundle count and total bytes across all four delete paths. Previously some paths left the counters stale. **Security:** *SSRF coverage for the full CGNAT range* — The check for `100.64.0.0/10` now uses a numeric comparison so the entire carrier-grade NAT range is blocked, not just the prefixes that happened to match the prior string test. · *Webhook DNS pinning closes a TOCTOU rebind* — The resolved IP from the SSRF check is reused for the outbound connection so a hostile DNS server cannot rebind between resolution and connect. · *CORS wildcard no longer advertises credentials* — When the configured origin is `*`, the response omits `Access-Control-Allow-Credentials`. This restores compliance with the Fetch spec, which forbids that combination. · *Vault files blocked from public share routes* — Public share routes refuse to serve vault file paths so unauthenticated callers cannot fetch sealed ciphertext through the share surface. · *Admin logo upload validates image magic bytes* — Uploaded logos must match an image signature; non-image content is rejected before it reaches disk. · *Customer Stash hardening* — Password-session check enforced on every upload route under `/stash/:slug`, accentColor is hex-validated, logoUrl is scheme-validated, and the per-user storage quota is enforced on the stash path. **References:** [FIPS 203 (ML-KEM)](https://csrc.nist.gov/pubs/fips/203/final) · [RFC 9106 (Argon2)](https://datatracker.ietf.org/doc/rfc9106/)
