# v1.9.0 — Opt-in Passphrase Wrapping for `vault.key`

**Status:** Design spec. Not yet implemented. Target release: v1.9.0.

**Context:** [`THREAT_MODEL.md`](../THREAT_MODEL.md) §9 L2 documents that `data/vault.key` is plaintext JSON protected only by filesystem permissions. This spec proposes an opt-in layer that wraps the vault key with an Argon2id-derived XChaCha20-Poly1305 encryption, closing the disk-snapshot threat vector for operators who choose to enable it.

**Scope of this release:** the wrapping format, the migration tooling, and the boot-time state machine that consumes both plaintext and wrapped formats. Rotation (`v1.9.1`), compromise UX (`v1.9.2`), and full vault key rotation (`v1.9.3`) are deferred and scoped in their own specs.

---

## 1. Non-negotiable principles

These constrain every decision in the rest of the spec:

1. **Zero regression.** A v1.8.26 → v1.9.0 upgrade with no configuration change must behave identically to v1.8.26. Existing `data/vault.key` continues to work, server boots silently, nothing in the admin UI changes.
2. **Downgrade must be possible.** Operators who enable then regret it must be able to return to plaintext `vault.key` without data loss. A removal tool is part of this release.
3. **Crash-safe migration.** At every point during setup, rotation, or removal, if the process is killed the vault is in a well-defined state — either fully migrated or fully not. Partial states that could confuse subsequent boots are explicitly detected and either auto-recovered or aborted with clear instructions.
4. **Loss of passphrase = loss of data.** There is no recovery mechanism. The setup tool must obtain explicit operator confirmation. Every doc surface references this prominently.
5. **Passphrase never touches disk** (beyond `VAULT_PASSPHRASE_FILE` when operator chooses that path), **never touches logs**, **never appears in stack traces or admin-visible env dumps**, **never appears in error messages** beyond "rejected / accepted" signals.
6. **No cloud or external dependencies.** Passphrase sources are env, file, or stdin. No AWS / GCP / HashiCorp Vault / TPM / HSM in this release. Future releases may add a driver abstraction if warranted; this release intentionally stays local-only.

## 2. On-disk file format — `data/vault.key.sealed`

Binary, self-describing, AEAD-authenticated. Magic byte `0xE2` distinguishes from the storage envelope (`0xE1`) and vault-sealed strings (`vault:` prefix).

### Layout

```
Offset  Field                       Size    Value / Range
──────  ──────────────────────────  ────    ──────────────────────────
 0      Magic                       1       0xE2
 1      Format version              1       0x01
 2      KDF algorithm ID            1       0x01 = Argon2id
 3      Reserved                    1       0x00
 4      Argon2 memory (KiB)         4       uint32 BE — 65536 default (64 MiB)
 8      Argon2 time cost            2       uint16 BE — 3 default
10      Argon2 parallelism          1       uint8   — 4 default
11      Salt length                 1       uint8   — 16 default
12      Salt                        VAR     random per seal/rotation
 N      Cipher algorithm ID         1       0x02 = XChaCha20-Poly1305
 N+1    Nonce length                1       uint8   — 24
 N+2    Nonce                       24      random per seal/rotation
 N+26   Ciphertext length           4       uint32 BE
 N+30   Ciphertext + Poly1305 tag   VAR     encrypted vault.key JSON bytes
```

### AAD binding

**Bytes 0 through (N+29) — everything before the ciphertext — are bound as AAD on the AEAD tag.** This includes the magic, version, KDF params, salt, cipher ID, nonce length, and nonce.

Rationale: an attacker who flipped e.g. the Argon2 memory cost from `65536` to `1` (trivially brute-forceable) and offered the modified file as legitimate would be detected — Poly1305 authentication fails because the AAD differs. Without header-as-AAD, downgrade-in-header attacks become a real risk when multiple KDFs or ciphers are eventually supported.

### Parameters in-header

Argon2 cost parameters live in the header rather than being hardcoded. Consequences:

- A 2028 operator can re-wrap with higher costs without a code change
- Existing files remain readable because their parameters travel with them
- A reviewer can inspect the header and see exactly what cost was used
- Rotation (v1.9.1) can upgrade parameters naturally

### Defaults (enforced by the setup tool)

| Parameter | Value | Rationale |
|---|---|---|
| Argon2 variant | `argon2id` (0x02) | Hybrid, recommended for password hashing by RFC 9106 |
| Memory cost | 65,536 KiB (64 MiB) | Matches server's `hashPassword()` — already tested on minimal hosts |
| Time cost | 3 | Matches server's `hashPassword()` |
| Parallelism | 4 | Matches server's `hashPassword()` |
| Salt length | 16 bytes | Standard for password-derived KDFs |
| Nonce length | 24 bytes | XChaCha20-Poly1305's full extended nonce |
| Cipher | XChaCha20-Poly1305 | Project-wide default |

### Magic byte registry

| Magic | Format | Used by |
|---|---|---|
| `0xE1` | Storage envelope | `lib/crypto.js` `encrypt()` / `decrypt()` |
| `0xE2` | Wrapped vault key | This spec |

Future wrapped-format magic bytes should continue in the `0xE?` range to keep "wrapped key material" visually distinct from other binary blobs.

## 3. Boot-time state machine (`lib/vault.js`)

The correctness story lives in this dispatch. Every boot evaluates these inputs:

| Input | Source |
|---|---|
| `A` | `data/vault.key` exists? |
| `B` | `data/vault.key.sealed` exists? |
| `C` | `data/vault.key.sealed.tmp` exists? (orphan) |
| `D` | `data/vault.key.migration-pending` exists? (in-progress marker) |
| `M` | `VAULT_PASSPHRASE_MODE` env var: `disabled` (default) or `required` |

### 3.1 Pre-dispatch cleanup

Before evaluating the main dispatch:

**If `C` (orphan `.tmp`) exists:**

```
fs.unlinkSync("data/vault.key.sealed.tmp")
log.info("Cleaned orphan vault.key.sealed.tmp from a previous crash")
```

Tmp files from `vault.key.sealed.tmp` are always orphans. The seal operation only writes to `.tmp` before renaming; a crashed tmp is never useful and is always safe to delete.

### 3.2 Migration marker recovery

**If `D` exists** (migration was in progress when last run crashed):

```
marker = JSON.parse(fs.readFileSync(migration_pending_path))
assert marker.format === 1
```

Then dispatch recovery:

| State | Action |
|---|---|
| Sealed file does NOT exist | Migration crashed before rename. Delete marker. Proceed with normal dispatch (which will see `A=yes, B=no`). |
| Sealed file's SHA3-512 ≠ `marker.sealedSha3` | Sealed file was replaced/corrupted between crash and restart. **Abort with explicit error** — operator must investigate. |
| `A` (plaintext) still exists | Crash between sealed-rename and plaintext-unlink. Attempt `fs.unlinkSync("data/vault.key")`. Delete marker. Proceed. |
| `A` does not exist | Crash between plaintext-unlink and marker-delete. Just delete the marker. Proceed. |

After recovery, the marker no longer exists and normal dispatch runs on the cleaned state.

### 3.3 Dispatch table

| A | B | M | Action |
|---|---|---|--------|
| Yes | No | `disabled` | **Current v1.8 behavior.** Load plaintext keys. |
| Yes | No | `required` | **Config mismatch — abort.** `"vault.key is plaintext but VAULT_PASSPHRASE_MODE=required. Run node scripts/vault-passphrase-setup.js to migrate, or unset the env var."` Exit 1. |
| No | Yes | `disabled` | **Config mismatch — abort.** `"vault.key.sealed exists but VAULT_PASSPHRASE_MODE is disabled. Either set it to required, or run node scripts/vault-passphrase-remove.js to unseal."` Exit 1. |
| No | Yes | `required` | **Normal wrapped boot.** Obtain passphrase from source driver. Unwrap. On failure: exit 1 with `"passphrase rejected"`. No retries, no partial state. |
| No | No | `disabled` | **First run, plaintext.** Generate keypair, write `vault.key`. Current behavior. |
| No | No | `required` | **First run, wrapped.** Obtain passphrase. Generate keypair in memory. Wrap. Write `vault.key.sealed`. Never write plaintext. |
| Yes | Yes | * | **Invariant violation — abort.** `"Both vault.key and vault.key.sealed exist. This indicates an incomplete migration or manual interference. Resolve by deleting the file you do NOT want to keep, then restart."` Exit 1. |

The **abort on ambiguity** policy is the central safety mechanism. The server refuses to guess what the operator intends when state is ambiguous, because guessing wrong would generate a new keypair and destroy all encrypted data.

### 3.4 Error-message content policy

Error messages on passphrase rejection **must not** include:
- The passphrase (obviously)
- The length of the attempted passphrase
- Whether the format header parsed successfully vs AEAD failed
- Any partial bytes of the key material

Allowed information:
- `"passphrase rejected"`
- `"vault.key.sealed appears corrupted — magic byte mismatch"` (before any crypto operations)
- `"unsupported vault.key.sealed format version N — upgrade HermitStash to a version that supports it"`

## 4. Setup tool — `scripts/vault-passphrase-setup.js`

Migrates an existing plaintext `data/vault.key` into the wrapped format and **deletes the plaintext** atomically. Opt out of the auto-delete with `--keep-plaintext` for operators who prefer a manual rollback window.

### 4.1 Pre-flight checks

Abort with clear message if any fail:

1. `data/vault.key` must exist (source material)
2. `data/vault.key.sealed` must NOT exist (refuse to overwrite)
3. `data/vault.key.sealed.tmp` must NOT exist (stale)
4. `data/vault.key.migration-pending` must NOT exist (previous crash — investigate before re-running)
5. Server must not respond on `http://127.0.0.1:3000/health` (best-effort detection; override with `--force-with-server-running`)
6. Refuse to run as UID 0 (general hygiene; override with `--allow-root`)

### 4.2 Passphrase sourcing

The tool uses the **same `lib/passphrase-source.js` driver the server uses.** This is important: it means the operator can test the exact source configuration they'll use in production against the wrapped file before the tool finalizes.

Priority (when `VAULT_PASSPHRASE_SOURCE` is unset or `auto`):

1. `VAULT_PASSPHRASE_FILE` — if set, read the file (trim trailing newlines)
2. `VAULT_PASSPHRASE` — if set, read env directly
3. Interactive stdin — if TTY is available, prompt

If none available: error out.

### 4.3 Interactive-stdin additions

When the source is stdin (no env/file configured):

1. Prompt for passphrase (hidden input)
2. Prompt for confirmation (hidden input), must match
3. Display a strength indicator (length + character class diversity — informational, not gating)
4. Display the warning:

   ```
   ⚠  Loss of this passphrase means loss of all encrypted data.
       HermitStash has no recovery mechanism.
       Your vault key, all user data, all files, all settings depend on this passphrase.
   ```

5. Require typed confirmation:

   ```
   Have you stored this passphrase in a password manager or equivalent safe location?
   Type YES (all caps) to confirm and proceed with auto-delete of data/vault.key:
   ```

   Any response other than literal `YES` aborts with no changes.

### 4.4 Execution sequence

Atomic, crash-recoverable. Each step's crash consequences are detailed in §9.

```
1. plaintext_json = fs.readFileSync("data/vault.key")

2. Derive:
     salt     = random(16)
     nonce    = random(24)
     wrap_key = Argon2id(passphrase, salt, m=65536, t=3, p=4, out_len=32)

3. Seal:
     header     = assemble_header(salt, nonce, argon_params, version=1)
     ciphertext = XChaCha20Poly1305.encrypt(
                    key=wrap_key, nonce=nonce,
                    plaintext=plaintext_json, aad=header)
     sealed_blob = header || ciphertext

4. Write durably:
     fs.writeFileSync("data/vault.key.sealed.tmp", sealed_blob, { mode: 0o600 })
     fsyncSync(".tmp")
     fsyncSync(data_directory)

5. In-process round-trip verify:
     read .tmp back from disk
     parse header, derive wrap_key again from same passphrase source
     decrypt, assert plaintext_json_decrypted === plaintext_json (byte-exact)
     On failure: unlink .tmp, abort with error. vault.key is UNTOUCHED.

6. Write migration marker:
     marker_json = {
       format: 1,
       startedAt: new Date().toISOString(),
       sealedSha3: sha3_512(sealed_blob).toString("hex")
     }
     fs.writeFileSync("data/vault.key.migration-pending.tmp", marker_json, { mode: 0o600 })
     fsyncSync(".tmp")
     fs.renameSync(".tmp", "data/vault.key.migration-pending")
     fsyncSync(data_directory)

7. Atomic rename sealed file into place:
     fs.renameSync("data/vault.key.sealed.tmp", "data/vault.key.sealed")
     fsyncSync(data_directory)

8. Delete plaintext (unless --keep-plaintext):
     fs.unlinkSync("data/vault.key")
     fsyncSync(data_directory)

9. Delete migration marker:
     fs.unlinkSync("data/vault.key.migration-pending")
     fsyncSync(data_directory)
```

### 4.5 Final operator output

```
✓ Migration complete.

   data/vault.key        — deleted
   data/vault.key.sealed — active (wrapped with passphrase)

To activate passphrase-protected boot:
  1. Set VAULT_PASSPHRASE_MODE=required in your environment.
  2. Ensure the passphrase source is configured the same way you configured it for this tool:
        VAULT_PASSPHRASE=<passphrase>              (env var)
        VAULT_PASSPHRASE_FILE=/path/to/secret      (Docker secrets style)
        VAULT_PASSPHRASE_SOURCE=stdin              (interactive prompt on boot)
  3. Restart the server. Expected output:
        [vault] Unsealing vault.key.sealed...
        [vault] Unsealed successfully.
  4. Log in and verify normal operation.

If anything goes wrong and you need to roll back:
  - Stop the server.
  - Run: node scripts/vault-passphrase-remove.js
  - Unset VAULT_PASSPHRASE_MODE.
  - Restart.
```

For `--keep-plaintext`, the output is adjusted to note that `vault.key` still exists as a backup and recommend manual deletion once confidence is established.

## 5. Removal tool — `scripts/vault-passphrase-remove.js`

Reverses setup. Unwraps `data/vault.key.sealed` back to `data/vault.key`. Used for:

- Operators reverting a decision
- Pre-downgrade to v1.8.x (which doesn't understand the wrapped format)
- Troubleshooting when a passphrase is suspected incorrect

### 5.1 Pre-flight

1. `data/vault.key.sealed` must exist (source)
2. `data/vault.key` must NOT exist (refuse to overwrite)
3. `data/vault.key.sealed.tmp` and `data/vault.key.migration-pending` must NOT exist
4. Server must not respond on `/health` (best-effort, overridable)

### 5.2 Execution

Same crash-recovery pattern as setup, in reverse. Uses a `vault.key.unseal-pending` marker analogous to `migration-pending`. Writes `vault.key.tmp`, fsync, writes marker, fsync, rename `.tmp` → `vault.key`, fsync, delete sealed file, fsync, delete marker, fsync.

The boot-time recovery logic (§3.2) handles this marker symmetrically.

### 5.3 Final output

```
✓ Unsealing complete.

   data/vault.key.sealed — deleted
   data/vault.key        — active (plaintext, filesystem-protected)

Remember to unset VAULT_PASSPHRASE_MODE (or set to `disabled`) before restarting
the server, otherwise boot will abort because plaintext + required is a config
mismatch.
```

## 6. Passphrase source drivers — `lib/passphrase-source.js`

Single interface, three drivers. Selected by `VAULT_PASSPHRASE_SOURCE` env var.

### 6.1 Interface

```javascript
module.exports = {
  async getPassphrase(): Promise<string>,
  sourceKind(): 'env' | 'file' | 'stdin',
};
```

### 6.2 Source selection

| `VAULT_PASSPHRASE_SOURCE` | Behavior |
|---|---|
| (unset) or `auto` | Priority order: `VAULT_PASSPHRASE_FILE` → `VAULT_PASSPHRASE` env → stdin (if TTY). Error if none available. |
| `file` | Must have `VAULT_PASSPHRASE_FILE` set. Error otherwise. |
| `env` | Must have `VAULT_PASSPHRASE` set and non-empty. Error otherwise. |
| `stdin` | Must have TTY on stdin. Error otherwise (useful for failing fast in headless Docker without `-it`). |

### 6.3 Driver semantics

**env:**
- Read `process.env.VAULT_PASSPHRASE`
- Reject if undefined, null, or empty string
- After read, `delete process.env.VAULT_PASSPHRASE` to limit exposure to later env-dump surfaces

**file:**
- Read `process.env.VAULT_PASSPHRASE_FILE` path
- `fs.readFileSync()` and `toString('utf8')`
- Trim trailing `\r\n`, `\n`, and `\r` (but NOT internal whitespace — passphrase may contain spaces)
- Reject if file is empty after trim
- Close immediately; do NOT cache

**stdin:**
- Use Node's `readline` with `createInterface({ terminal: true })` and manually suppress echo via `process.stdin.setRawMode(true)`
- Display prompt `"Vault passphrase: "`
- Read until newline, restore raw mode
- On CTRL-C: exit 1 with no side effects

### 6.4 Passphrase validation (constraints, not policy)

- Minimum length: **no enforcement**. Operator's choice. An empty passphrase is rejected at the source layer (see above); anything non-empty passes.
- Maximum length: 4,096 bytes. Argon2id handles arbitrary length, but a multi-MB passphrase is suspicious and likely a misconfiguration.
- Character set: bytes, not code points. UTF-8, ASCII, binary — all accepted.

Length / character-class **strength indicators** in the setup tool are informational only, not gating. Operator may choose a short passphrase knowingly; the tool records the choice in its audit-log message but does not refuse.

## 7. Redaction surfaces

Treat `VAULT_PASSPHRASE` and `VAULT_PASSPHRASE_FILE` as sensitive values in the following surfaces:

### 7.1 Admin environment endpoint

`routes/admin.js` `GET /admin/environment`: currently returns the full `process.env` snapshot. Update to redact:

```javascript
const SENSITIVE_ENV_KEYS = [
  'VAULT_PASSPHRASE',
  'VAULT_PASSPHRASE_FILE',
  // ... existing sensitive keys
];

function redactEnv(env) {
  const out = { ...env };
  for (const k of SENSITIVE_ENV_KEYS) if (k in out) out[k] = '***';
  return out;
}
```

`VAULT_PASSPHRASE_MODE` is **not** redacted — operator needs visibility into whether protection is on.

### 7.2 Logger

`app/shared/logger.js` already has a sensitive-key filter. Add `VAULT_PASSPHRASE*` patterns to it.

### 7.3 Error handler

`middleware/error-handler.js`: verify no env dump in 500 responses (already true; add a regression test).

### 7.4 Audit log

`audit.log()` calls related to vault operations must never include passphrase material in the `details` field. Audit events for vault setup / rotation / removal should log only:

- Which operation (`VAULT_PASSPHRASE_ENABLED`, `VAULT_PASSPHRASE_REMOVED`)
- When
- Which admin user initiated (if via future UI)
- Argon2 parameters used (safe to log — they're in the sealed header anyway)

## 9. Edge cases catalogued

### 9.1 Normal operation

| Case | Handling |
|---|---|
| Wrong passphrase on boot | AEAD tag mismatch → log "passphrase rejected", exit 1. No retries. No partial state. |
| Empty passphrase | Setup tool: rejected at prompt. Boot: passphrase source returns empty → error before any crypto operation. |
| Binary / non-UTF8 passphrase | Accepted. Argon2 takes bytes. File source: read raw bytes, don't UTF-8 decode. |
| Passphrase > 4096 bytes | Rejected. Sanity limit. |
| Argon2 params in header request > available RAM | `wrap()` / `unwrap()` may OOM. Setup tool validates against `os.freemem()` before starting. Boot can't predict host RAM at wrap time so fails hard with a clear error. |
| Argon2 params in header are unusually low (1 KiB memory) | Accepted during unwrap — operator may have had reasons. Setup tool rejects writing such values unless `--force-weak-params` passed with an explicit warning. |
| File source: file doesn't end with newline | No problem, trim is idempotent. |
| File source: file has trailing `\r\n` | Trimmed. Accepts both Unix and Windows line endings. |
| File source: file has leading whitespace | Accepted as part of passphrase. Operator may have intended it. |
| Stdin source: no TTY (headless Docker) | Setup tool errors with `"stdin source requires a TTY (run with docker run -it)"`. Boot errors with same message. |
| Stdin source: CTRL-C during prompt | Exit 1. Raw mode restored via process exit handler. No side effects. |

### 9.2 Crash recovery during setup

Setup execution numbered steps per §4.4:

| Crash point | State after crash | Recovery |
|---|---|---|
| Before step 4 | `vault.key` exists, no `.tmp`, no marker | Nothing to recover. `vault.key` intact. Operator re-runs setup. |
| After step 4, before step 6 | `vault.key` + `.tmp` exist, no marker | Boot cleanup unlinks `.tmp`. `vault.key` intact. Operator re-runs setup. |
| After step 6, before step 7 | `vault.key` + `.tmp` + marker all exist | Boot sees marker but sealed file (post-rename) doesn't exist. Marker's `sealedSha3` refers to the `.tmp`. Recovery aborts with `"marker indicates incomplete migration. Delete data/vault.key.sealed.tmp and data/vault.key.migration-pending, then re-run setup."` |
| After step 7, before step 8 | `vault.key` + sealed + marker all exist | Boot recovery: verify `sha3(sealed)` matches marker's hash, unlink `vault.key`, unlink marker, proceed. Server boots wrapped. |
| After step 8, before step 9 | sealed + marker exist, plaintext gone | Boot recovery: verify hash match, unlink marker, proceed. Server boots wrapped. |
| After step 9 | sealed only | Normal state. Server boots wrapped. |

### 9.3 Tampering / adversarial states

| State | Handling |
|---|---|
| Marker file with tampered `sealedSha3` | Hash mismatch during recovery → abort with `"vault.key.sealed hash does not match migration marker. Possible tampering. Investigate manually before restarting."` |
| Marker file with unknown `format` field | Abort with `"unknown migration marker format. Upgrade HermitStash or manually resolve."` |
| Sealed file replaced with arbitrary bytes | Unwrap fails at AEAD verification. Operator cannot decrypt. Treated as "passphrase rejected" behavior — data is lost if no backup. |
| Sealed file magic byte wrong | Unwrap rejects at header parse. Clearer error: `"vault.key.sealed is not a valid wrapped vault file (magic byte mismatch)"`. |

### 9.4 Operator mistakes

| Mistake | Outcome |
|---|---|
| Operator sets `VAULT_PASSPHRASE` to wrong value in prod after successful setup | Boot fails at AEAD verification. Operator discovers the mistake at restart, fixes env, retries. No data loss (sealed file intact). |
| Operator deletes `vault.key.sealed` while `VAULT_PASSPHRASE_MODE=required` | Boot falls into `A=no, B=no, M=required` branch: "First run, wrapped." Generates NEW keypair and asks for a passphrase to wrap it. **The old data is now unrecoverable.** This is unavoidable without a backup — we cannot distinguish "first run" from "file deleted." |
| Operator runs setup tool twice in a row | Second run pre-flight: `data/vault.key.sealed` exists → abort with clear message. No changes. |
| Operator runs remove tool with wrong passphrase | `passphrase-source` returns the wrong passphrase → AEAD verification fails during unwrap → tool aborts with "passphrase rejected" before writing anything. |
| Operator runs setup with `--keep-plaintext` then later forgets to delete `vault.key` | Boot sees both files (A=yes, B=yes) → invariant violation abort. Operator decides manually. |
| Operator downgrades v1.9.x to v1.8.x with only `vault.key.sealed` present | v1.8.x doesn't know the format. Sees `A=no` → generates new keypair → **data loss**. Documented prominently; use the removal tool BEFORE downgrading. |

## 10. Testing plan

### 10.1 Unit tests (`tests/test-vault-wrap.js`)

- Round-trip: `wrap(x, pw)` then `unwrap(result, pw)` returns `x` byte-exact
- Round-trip with every pairing of supported parameters (memory/time/parallelism grid)
- Round-trip with binary plaintext (random bytes, JSON with unicode, empty string edge case)
- Wrong passphrase: `unwrap(result, "wrong")` throws, doesn't leak partial plaintext
- Header corruption (flip each byte of header) → AEAD tag failure or header parse failure, never successful bogus decrypt
- Unknown magic byte → parse error before any crypto operation
- Unknown format version → explicit "unsupported version" error
- Unknown KDF / cipher ID → explicit "unsupported" error
- 1000 seals produce 1000 unique salts and 1000 unique nonces (no RNG starvation)
- Known-answer test vectors: seed the RNG, verify deterministic output for snapshot regression

### 10.2 State machine tests (`tests/test-vault-boot.js`)

Each of the 7 dispatch table rows (§3.3), verified by:
- Setting up the filesystem state
- Setting env vars
- Calling `loadKeys()` in isolation
- Asserting the expected outcome (keys loaded, or specific error message, or exit code)

Additionally:
- Orphan `.tmp` cleanup: create `.tmp`, boot, assert it's gone
- Marker recovery: each of the 6 crash-point rows from §9.2, synthesized as filesystem state

### 10.3 Setup/remove integration tests (`tests/test-passphrase-setup-recovery.js`)

For each numbered step in §4.4:
- Start setup with known inputs
- Use a crash harness (spawn child process, kill -9 at the target step, boundary-detected via fs.watch or sync-point hooks)
- Restart the server
- Assert the expected post-crash state and recovery behavior

Additional scenarios:
- Setup happy path: start with plaintext, run setup, start server with `VAULT_PASSPHRASE_MODE=required`, verify full CRUD operations work identically to disabled mode
- Remove happy path: setup, then remove, byte-compare resulting `vault.key` to pre-setup bytes
- Setup with `--keep-plaintext`, then verify invariant-violation abort on boot

### 10.4 E2E under wrapped mode (`tests/test-passphrase-e2e.js`)

Run the full existing E2E suite (currently 405 tests) once with `VAULT_PASSPHRASE_MODE=required` and an environment-injected passphrase. Any regression vs disabled mode is a blocker.

### 10.5 Tampering tests (`tests/test-vault-wrap-tamper.js`)

- Bit-flip each byte of the header → AEAD rejection
- Bit-flip each byte of the ciphertext → AEAD rejection
- Truncate ciphertext → header parse error
- Extend ciphertext with trailing bytes → AEAD rejection or header length mismatch
- Replace salt → derives wrong key → AEAD rejection
- Replace nonce → AEAD rejection

### 10.6 Performance benchmarks

- `Argon2id(64 MiB, 3, 4)` derivation time on reference hardware: measure, log, compare to README claim
- Ensure boot time with wrapped mode is under 2 seconds on a reference small host (the memory cost is the main factor)

# v1.9.0 — Opt-in passphrase protection for vault.key

## Highlights

- **Opt-in passphrase wrapping.** A new optional layer wraps the at-rest
  vault keypair with Argon2id-derived XChaCha20-Poly1305 encryption.
  When enabled, an attacker with a disk snapshot but without the
  passphrase cannot recover the vault key — closing the threat gap
  documented in THREAT_MODEL.md §9 L2.
- **Zero regression.** Default is disabled. Existing deployments
  continue to work unchanged.
- **Deliberate, reversible migration.** A dedicated CLI tool migrates
  an existing plaintext vault.key to the wrapped format with in-process
  round-trip verification and crash-safe atomic file operations. A
  matching removal tool reverses the operation.
- **Automatic plaintext deletion after verified wrapping.** The setup
  tool verifies the wrapped file round-trips correctly, then atomically
  deletes the plaintext backup. Opt out with `--keep-plaintext` if you
  want a manual rollback window.

## How to enable

See the new "Passphrase protection" section in README.md, and the
detailed spec in docs/specs/v1.9.0-passphrase-wrapping.md.

## ⚠  Critical warning

Loss of the passphrase means loss of all encrypted data. HermitStash
has no recovery mechanism. Store the passphrase in a password manager
BEFORE running setup. The tool requires typed confirmation that you've
done so.

## Downgrade path

If you need to go back to v1.8.x:
  1. Run `node scripts/vault-passphrase-remove.js` under v1.9.x
     (unwraps back to plaintext)
  2. Unset VAULT_PASSPHRASE_MODE
  3. Downgrade

Never attempt a direct v1.9.x (sealed) → v1.8.x downgrade. v1.8.x
does not understand the wrapped format and would generate a new
keypair, losing access to all encrypted data.

## 13. Open questions

Before implementation starts, the following are worth nailing down:

1. **Marker file format stability.** The `sealedSha3` field binds the marker to a specific sealed file. Is there a scenario where we'd want to change the hash algorithm later? If yes, the marker should carry an `alg` field. Recommended: include `hashAlg: "sha3-512"` in the marker JSON now, so future changes are compatible.

2. **`--keep-plaintext` default.** Should the *default* behavior keep the plaintext (safer for first adopters) or delete it (cleaner UX)? Current spec says "delete by default, opt out with flag." An alternative is "keep by default, explicit `--auto-delete` to opt in." The chosen default should match the tone of other destructive HermitStash operations — most of which require explicit typed confirmation rather than flag opt-in.

3. **Argon2 parameter self-upgrade on boot.** If an operator later rotates their passphrase with higher Argon2 costs via the v1.9.1 rotation tool, should v1.9.3+ offer a "your parameters are out of date, rotate to current defaults?" nudge? This is a cosmetic question but worth deciding before building rotation.

4. **Boot-time Argon2 progress reporting.** 64 MiB Argon2id takes ~500ms-1s on typical hardware. Should the boot sequence log "Deriving wrapping key..." at the start of the unwrap operation? Good for ops visibility; bad if logs are copied to an observability system that records the event-rate.

5. **Rate limiting on setup/remove tools.** Running the tool requires filesystem access (not a network surface), so traditional rate limiting doesn't apply. However, an attacker with write access to `data/` could repeatedly test passphrases against `vault.key.sealed` offline. Mitigation: Argon2's memory cost inherently rate-limits this. Current default of 64 MiB, 3 iterations means ~1s per attempt — strong enough for offline resistance if the passphrase has reasonable entropy. Document this in the threat model.
